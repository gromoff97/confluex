# AGENTS

- READ ONLY THE RELEVANT `FR-<AREA>.md`.
- IF WORK SPANS AREAS, READ EVERY RELEVANT `FR-<AREA>.md`.
- AREA MAP: `BASE`=SHARED PRIMITIVES AND COMMON SERIALIZATION; `CMD`=COMMAND SURFACE; `UX`=OPERATOR EXPERIENCE; `VAL`=INVOCATION VALIDATION; `OPT`=OPTION SEMANTICS; `DIAG`=DIAGNOSTICS; `CONF`=CONFIGURATION; `LIFE`=INSTALLATION LIFECYCLE; `RUN`=RUN LIFECYCLE; `SCOPE`=SCOPE DISCOVERY; `DATA`=DATA ACQUISITION; `OUT`=OUTPUTS AND ARTIFACTS; `REP`=REPORTS; `SAFE`=SAFETY; `INT`=INTERRUPTION AND RUNTIME FAILURE; `RES`=RESUME AND RECOVERY; `OBS`=OBSERVABILITY AND OUTCOMES.
- NEW OR CHANGED REQUIREMENTS MUST BE SELF-CONTAINED, UNAMBIGUOUS, CONFLICT-CHECKED AGAINST THE EXISTING CORPUS, AND DIRECTLY TESTABLE FROM THE CARD TEXT.
- USE STABLE `FR-<NNN>` IDS. DO NOT RENUMBER OR REUSE THEM.
- ONE REQUIREMENT = ONE OBLIGATION.
- USE EXACT CARD SHAPE: `### FR-<NNN>`, `**Requirement**:`, `**Applicability**:`, `**Rationale**:`, `**Acceptance Criteria**:`, `**Dependencies**:`, `**Traceability**:`.
- REQUIREMENT TEXT AND ACCEPTANCE CRITERIA MUST BE SELF-CONTAINED, TESTABLE, AND FREE OF VAGUE TERMS UNLESS ANOTHER CARD DEFINES THEM PRECISELY.
- EVERY NORMATIVE TERM, CONTROLLED VOCABULARY, AND REUSED SERIALIZATION FORM USED IN REQUIREMENT TEXT OR ACCEPTANCE CRITERIA MUST BE DEFINED IN THE CORPUS BEFORE OR WHERE IT IS FIRST USED.
- ACCEPTANCE CRITERIA MUST DEFINE EXACT TOKENS, ORDERING, CASING, SERIALIZATION, DEFAULTS, AND ABSENCE SEMANTICS FOR MACHINE-CHECKABLE PARTS OF GOVERNED CLI TEXT, FILES, PATHS, SCHEMAS, AND STATUS VALUES. EXPLANATORY PROSE MAY REMAIN FREE-FORM ONLY WHEN THE CARD STILL DEFINES THE MANDATORY PREFIXES, TOKENS, SECTION ORDER, AND REQUIRED IDENTIFYING CONTENT.
- IF A SORT ORDER CAN TIE, ACCEPTANCE CRITERIA MUST DEFINE SUCCESSIVE TIE-BREAKERS UNTIL THE ORDER IS DETERMINISTIC.
- CROSS-REFERENCES MUST USE EXISTING `FR-<NNN>` IDS ONLY. DO NOT USE UNDEFINED SECTION REFERENCES OR UNSTATED EXTERNAL CONTEXT FOR NORMATIVE BEHAVIOR.
- EACH OBLIGATION MUST HAVE ONE AUTHORITATIVE HOME. OPTION CARDS OWN REQUEST/SELECTION SEMANTICS; OUTCOME, ARTIFACT, AND REPORT CARDS OWN RESULTING STATUS, FILE, AND SCHEMA CONTRACTS.
- SECONDARY CARDS MUST DEFER TO THE AUTHORITATIVE HOME FOR ARTIFACT PRESENCE, FILE SETS, AND SCHEMA SHAPE INSTEAD OF RESTATING THOSE DUTIES.
- OPTION CARDS MAY REQUEST PERSISTED-STATE CHANGES OR ARTIFACT-RETENTION MODES, BUT THEY DO NOT OWN THE RESULTING STATE MUTATION, READBACK CONTRACT, FILE PRESENCE, OR FILE LAYOUT WHEN ANOTHER CARD ALREADY OWNS THAT BEHAVIOR.
- CARDS THAT MENTION SUMMARY FIELDS, EXIT CODES, CLEANUP, OR RETAINED-ARTIFACT BRANCHES MUST EITHER OWN THOSE TOKENS AND BRANCH RULES AUTHORITATIVELY OR DEFER TO THE AUTHORITATIVE CARD WITHOUT REDEFINING THEM PIECEWISE.
- WHEN ONE SCENARIO CAN RETAIN MULTIPLE FINAL ARTIFACTS, ANY SINGULAR OUTPUT FIELD OR LINE THAT NAMES THE FINAL ARTIFACT MUST USE AN EXPLICIT PRECEDENCE RULE TO SELECT EXACTLY ONE AUTHORITATIVE ARTIFACT.
- WHEN ONE SCENARIO CAN END IN DIFFERENT RETENTION, CLEANUP, OR REPORT-PRESENCE BRANCHES, THE AUTHORITATIVE CARDS MUST DEFINE EACH BRANCH EXPLICITLY SO THAT SECONDARY CARDS CAN REUSE THOSE BRANCHES BY REFERENCE.
- SHARED SERIALIZATION PRIMITIVES SUCH AS PATH QUOTING, LIST DELIMITING, AND RELATIVE-PATH ENCODING MUST BE DEFINED ONCE AND THEN REUSED BY REFERENCE.
- REUSED ABSENCE TOKENS AND DELIMITED TOKEN-LIST FORMS ARE SHARED SERIALIZATION PRIMITIVES. THEY MUST HAVE ONE AUTHORITATIVE CARD AND MUST NOT BE INTRODUCED OR REDEFINED INCIDENTALLY INSIDE DOMAIN-SPECIFIC CARDS.
- DEPENDENCIES MUST LIST ONLY THE EXISTING CARDS MATERIALLY REQUIRED TO INTERPRET THE REQUIREMENT OR VERIFY ITS ACCEPTANCE CRITERIA. THEY ARE NOT A GENERIC RELATED-CARDS INDEX.
- IF ACCEPTANCE CRITERIA NAME SUMMARY FIELDS, EXIT CODES, RETAINED ARTIFACTS, CLEANUP BEHAVIOR, OR PERSISTED-STATE EFFECTS, DEPENDENCIES MUST INCLUDE THE AUTHORITATIVE CARDS THAT DEFINE THOSE TOKENS OR BRANCHES.
- MATERIAL REQUIREMENT CHANGES MUST UPDATE ACCEPTANCE CRITERIA, DEPENDENCIES, AND TRACEABILITY.
- IF REQUIREMENT CHANGES AFFECT CODE OR TESTS, APPLY ROOT `AGENTS.md` REVIEW CHECKLIST.
- DO NOT DUPLICATE REQUIREMENTS ACROSS FILES.
- DO NOT PUT PRODUCT REQUIREMENTS IN ANY `AGENTS.md`.
