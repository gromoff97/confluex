# AGENTS

- READ ONLY THE RELEVANT `FR-<AREA>.md`.
- IF WORK SPANS AREAS, READ EVERY RELEVANT `FR-<AREA>.md`.
- AREA MAP: `CMD`=COMMAND SURFACE; `UX`=OPERATOR EXPERIENCE; `VAL`=INVOCATION VALIDATION; `OPT`=OPTION SEMANTICS; `DIAG`=DIAGNOSTICS; `CONF`=CONFIGURATION; `LIFE`=INSTALLATION LIFECYCLE; `RUN`=RUN LIFECYCLE; `SCOPE`=SCOPE DISCOVERY; `DATA`=DATA ACQUISITION; `OUT`=OUTPUTS AND ARTIFACTS; `REP`=REPORTS; `SAFE`=SAFETY; `INT`=INTERRUPTION AND RUNTIME FAILURE; `RES`=RESUME AND RECOVERY; `SEC`=ENCRYPTION; `OBS`=OBSERVABILITY AND OUTCOMES.
- NEW OR CHANGED REQUIREMENTS MUST SATISFY CIRCUS MATTA.
- USE STABLE `FR-<NNN>` IDS. DO NOT RENUMBER OR REUSE THEM.
- ONE REQUIREMENT = ONE OBLIGATION.
- USE EXACT CARD SHAPE: `### FR-<NNN>`, `**Requirement**:`, `**Applicability**:`, `**Rationale**:`, `**Acceptance Criteria**:`, `**Dependencies**:`, `**Traceability**:`.
- REQUIREMENT TEXT AND ACCEPTANCE CRITERIA MUST BE SELF-CONTAINED, TESTABLE, AND FREE OF VAGUE TERMS UNLESS ANOTHER CARD DEFINES THEM PRECISELY.
- ACCEPTANCE CRITERIA MUST DEFINE EXACT TOKENS, ORDERING, CASING, SERIALIZATION, DEFAULTS, AND ABSENCE SEMANTICS WHEN THEY GOVERN CLI TEXT, FILES, PATHS, SCHEMAS, OR STATUS VALUES.
- CROSS-REFERENCES MUST USE EXISTING `FR-<NNN>` IDS ONLY. DO NOT USE UNDEFINED SECTION REFERENCES OR UNSTATED EXTERNAL CONTEXT FOR NORMATIVE BEHAVIOR.
- EACH OBLIGATION MUST HAVE ONE AUTHORITATIVE HOME. OPTION CARDS OWN REQUEST/SELECTION SEMANTICS; OUTCOME, ARTIFACT, AND REPORT CARDS OWN RESULTING STATUS, FILE, AND SCHEMA CONTRACTS.
- DEPENDENCIES MUST LIST ONLY THE EXISTING CARDS MATERIALLY REQUIRED TO INTERPRET OR VERIFY THE REQUIREMENT.
- MATERIAL REQUIREMENT CHANGES MUST UPDATE ACCEPTANCE CRITERIA, DEPENDENCIES, AND TRACEABILITY.
- IF REQUIREMENT CHANGES AFFECT CODE OR TESTS, APPLY ROOT `AGENTS.md` REVIEW CHECKLIST.
- DO NOT DUPLICATE REQUIREMENTS ACROSS FILES.
- DO NOT PUT PRODUCT REQUIREMENTS IN ANY `AGENTS.md`.
